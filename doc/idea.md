0단계. 기준선 고정

이 단계에서는 코드를 많이 짜는 게 아니라, 앞으로 안 흔들릴 최소 규격을 정한다.

정해야 하는 것:

월드 tick 단위
백엔드/프론트/에이전트 서비스 분리 방식
기본 행동 4종: 이동, 공격, 발화, 사용
캐릭터 상태 스키마
맵/타일/엔티티 상태 스키마
이벤트 메시지 형식
생각 파일 / 소울 파일 기본 포맷

여기서 중요한 건 “멋지게”가 아니라 앞으로 계속 쓸 공용 데이터 구조를 고정하는 것이다.

완료 기준:

캐릭터 상태 JSON이 정해져 있음
행동 요청/결과 JSON이 정해져 있음
생각 파일과 소울 파일의 텍스트 구조가 정해져 있음
1단계. 룰 엔진만 있는 최소 게임 백엔드

여기서는 에이전트 붙이지 말고, 먼저 게임이 세계를 제대로 판정하는지부터 만든다.

구현 범위:

타일맵 로드
구조물/장애물 배치
엔티티 배치
위치 이동 판정
공격 가능 여부와 간단한 데미지 처리
발화 범위 판정
아이템 사용 처리
기본 상태 변화 처리
시간 경과 처리
배고픔/스태미나 자연 감소

이 단계에서는 사람이 직접 API를 호출해서 테스트해도 된다.

예:

/action 으로 이동 요청
성공 시 위치 변경
실패 시 blocked 리턴
사용 시 hunger 감소
공격 시 hp 감소

중요한 건 이때부터 결과를 백엔드가 절대 기준으로 계산해야 한다는 점이다.

완료 기준:

에이전트 없이도 캐릭터 하나를 움직이고 공격/발화/사용 가능
성공/실패 결과와 상태 변화가 서버에서 정확히 계산됨
프론트 없이도 테스트 가능
2단계. 원시 이벤트 로그 추가

이 단계에서 로깅을 붙인다.
네가 말한 대로 이건 필수다.

저장할 최소 로그:

tick
timestamp
actor_id
action_type
target or position
result
fail_reason
state_delta
world_delta

예시 느낌:

{
  "tick": 183,
  "actor_id": "npc_01",
  "action": "USE",
  "target_id": "bread_02",
  "result": "success",
  "state_delta": {
    "hunger": -20,
    "gold": -5
  },
  "world_delta": {
    "removed_entities": ["bread_02"]
  }
}

이 단계의 목적은 디버깅과 나중의 수련 기반 만들기다.

완료 기준:

모든 행동이 원시 이벤트 로그로 남음
나중에 특정 행동 횟수/성공률 집계 가능
특정 NPC의 행동 히스토리 추적 가능
3단계. 프론트와 실시간 동기화

이제 프론트를 붙인다.
하지만 아직 에이전트는 붙이지 않는다.

구현 범위:

타일 렌더링
구조물 렌더링
엔티티 렌더링
위치 이동 반영
상태 반영
발화 말풍선이나 로그 표시
서버 이벤트 스트림 수신

여기서 중요한 건 프론트가 똑똑해지면 안 된다는 점이다.
프론트는 그리기와 보여주기만 해야 한다.

완료 기준:

서버 상태가 바뀌면 프론트에 즉시 반영됨
위치, 상태, 간단한 발화가 시각적으로 보임
서버 재접속 시 상태 복원 가능
4단계. 에이전트 없는 더미 NPC 루프

이 단계가 중요하다.
바로 SLM 붙이지 말고, 먼저 정해진 규칙대로 움직이는 더미 NPC를 넣는다.

예:

배고픔이 높으면 음식점 쪽으로 감
적이 가까우면 도망감
피곤하면 쉬는 장소로 감
사람이 근처면 간단히 발화함

이 단계의 이유는 간단하다.
에이전트 문제가 아니라 월드 구조 자체가 살아 있는지를 먼저 봐야 한다.

완료 기준:

NPC가 룰 기반으로 기본 생존 행동을 함
배고픔/스태미나 때문에 행동 방향이 바뀜
월드가 최소한의 생활감이 생김
5단계. 에이전트 입력 컨텍스트 파이프라인

이제 처음으로 LLM/SLM 쪽을 붙인다.
하지만 아직 기억 갱신은 안 한다. 먼저 입력 컨텍스트를 제대로 조립하는 것만 한다.

에이전트 입력에 들어갈 것:

현재 주변 환경
본인 상태
소울 파일
생각 파일
최근 행동 결과
가능 행동 목록

여기서 중요한 건 가능 행동 목록을 엔진 기준으로 정리해서 주는 것이다.

예:

{
  "available_actions": [
    {"type": "MOVE", "direction": "NORTH"},
    {"type": "MOVE", "direction": "WEST"},
    {"type": "TALK", "target_id": "merchant_01"},
    {"type": "USE", "target_id": "bread_02"}
  ]
}

이 단계에서는 모델이 자유롭게 소설 쓰지 않게, 반드시 정규화된 행동 하나만 내도록 해야 한다.

완료 기준:

백엔드 상태를 에이전트 프롬프트용으로 조립 가능
모델 출력이 고정된 행동 JSON으로 떨어짐
행동을 다시 서버에 넣어 실행 가능
6단계. 단일 NPC 에이전트 루프 완성

이제 NPC 1명에게만 진짜 루프를 돌린다.

루프:

상태 수집
입력 컨텍스트 조립
행동 결정
서버 실행
결과 수신
다음 루프 대기

트리거:

일정 텀마다
행동 결과 수신 시
큰 환경 변화 시

여기서 목표는 “똑똑함”이 아니다.
한 명이 최소한 자기 상태를 보고 생존형 행동을 반복하는지를 보는 거다.

완료 기준:

NPC 1명이 굶으면 음식 찾고
피곤하면 쉬고
근처 인간이나 상인에게 반응하고
적이 위험하면 피하려는 행동을 보임
7단계. 생각 파일 자동 갱신

이제 행동 후 정리 루프를 붙인다.
여기서부터 네 시스템의 핵심 감각이 살아난다.

입력:

방금 행동
행동 결과
최근 이벤트 몇 개
기존 생각 파일

출력:

갱신된 생각 파일

생각 파일은 하나의 문서로 유지한다.
권장 구조는 이 정도면 충분하다.

8단계. 장기 기억 저장

이제 장기 기억을 붙인다.
처음부터 복잡하게 하지 말고, 중요 사건만 에피소드로 저장하면 된다.

  "tags": ["market", "merchant", "money", "rejection"],
  "importance": 0.72
}

완료 기준:

행동 결과 중 일부가 장기 기억으로 축적됨
같은 사건이 중복 저장되지 않도록 최소 정리 규칙 있음
9단계. 장기 기억 검색 후 행동 반영

이제야 진짜 “기억을 활용하는 NPC”가 된다.

검색 방식은 2단계가 좋다.

1차 검색:

현재 상태/욕구 기반
예: 배고픔, 돈 부족, 위치, 시간대

2차 검색:

후보 행동 기반
예: 시장 상인에게 말 걸기
그러면 상인 관련 기억, 시장 거래 기억 재검색

이렇게 하면 검색 품질이 훨씬 좋다.

완료 기준:

기억이 행동 선택에 영향을 줌
같은 장소/사람/욕구 상황에서 과거 경험을 반영함
단순 랜덤 NPC가 아니라 경험 기반 NPC처럼 보임
10단계. NPC 3~5명으로 확장

이제 여러 명으로 늘린다.
하지만 여기서도 한 번에 대규모로 가면 안 된다.

먼저 넣을 것:

서로 다른 소울 파일
서로 다른 생각 파일
서로 다른 출발 인벤토리/돈/상태
동일한 월드 룰

여기서 확인할 것:

서로 다른 선택 패턴이 나오는가
같은 사건을 다른 식으로 기억하는가
자원 경쟁이 생기는가
사회적 분위기가 생기는가

완료 기준:

NPC마다 비슷한 상황에서도 다르게 행동함
월드가 더 이상 “복붙 캐릭터 군집”처럼 안 보임
